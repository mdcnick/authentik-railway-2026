## Directory Purposes

**Project Root:**
- Purpose: Container configuration and deployment orchestration
- Contains: Docker image definitions for multi-service setup
- Key files: `Dockerfile` (server), `worker/Dockerfile` (worker)

**worker/ Directory:**
- Purpose: Isolated configuration for background task processing service
- Contains: Dedicated Dockerfile for worker container
- Distinction: Separate from server to enable independent scaling and deployment

**.planning/codebase/ Directory:**
- Purpose: Generated architecture and structure analysis documents
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, STACK.md, INTEGRATIONS.md, CONCERNS.md
- Generated by: GSD codebase mapping tools

## Key File Locations

**Entry Points:**

- `Dockerfile`: Server service entry point with `CMD ["server"]`
- `worker/Dockerfile`: Worker service entry point with `CMD ["worker"]`

**Configuration:**

- Both Dockerfiles define build arguments (ARG): AUTHENTIK_REDIS__*, AUTHENTIK_POSTGRESQL__*, AUTHENTIK_SECRET_KEY, PORT
- All arguments converted to environment variables (ENV) for runtime access

**Infrastructure:**

- No explicit infrastructure-as-code files (assumes external orchestration tool like Railway, Docker Compose, or Kubernetes)
- Service dependencies configured via environment variables at container startup

## Naming Conventions

**Files:**

- `Dockerfile`: Standard Docker build file naming (no language-specific suffix)
- Capitalization: Capital `D` in Dockerfile (Docker standard)

**Directories:**

- Lowercase with hyphens for separating concerns (`worker/`)
- Functional naming (`worker` for background job processor)

**Environment Variables:**

- Pattern: `AUTHENTIK_{SYSTEM}__{PROPERTY}`
  - `AUTHENTIK_REDIS__HOST`, `AUTHENTIK_REDIS__PORT`, `AUTHENTIK_REDIS__PASSWORD`
  - `AUTHENTIK_POSTGRESQL__HOST`, `AUTHENTIK_POSTGRESQL__USER`, `AUTHENTIK_POSTGRESQL__NAME`, `AUTHENTIK_POSTGRESQL__PASSWORD`, `AUTHENTIK_POSTGRESQL__PORT`
  - `AUTHENTIK_SECRET_KEY`
  - `PORT` (application port)

## Where to Add New Code

**New Service/Container:**

- Create new directory at project root: `newservice/`
- Add `newservice/Dockerfile` with appropriate base image and service command
- Follow the ARG/ENV pattern for configuration injection
- Ensure service connects to shared PostgreSQL and Redis infrastructure

**Service-Specific Logic:**

- Server logic: Included in base image `ghcr.io/goauthentik/server:latest`
- Worker logic: Included in base image `ghcr.io/goauthentik/server:latest`
- No custom code modifications at this level — this is a deployment wrapper around upstream Authentik

**Configuration Additions:**

- Add new build argument (ARG) in both Dockerfiles if universally needed
- Service-specific argument: Add only to relevant Dockerfile (`worker/Dockerfile` or `Dockerfile`)
- Follow naming convention: `AUTHENTIK_COMPONENT__PROPERTY` or `COMPONENT__PROPERTY`

**Deployment Configuration:**

- Add to orchestration tool configuration (Railway, Docker Compose, Kubernetes manifests)
- Not managed in this repository — external system synchronizes environment variables to containers

## Special Directories

**Base Image Container:**

- Location: `ghcr.io/goauthentik/server:latest` (external)
- Source: Authentik project main repository (not in this codebase)
- Purpose: Contains all Authentik application logic
- Status: Pulled at container build time

**.git/ Directory:**

**.planning/ Directory:**

- Purpose: Codebase analysis and planning documentation
- Generated: Yes (generated by GSD mapping tools)
- Committed: Yes (should be committed for team reference)

## Service Architecture

**Horizontal vs. Vertical Scaling:**

- Multiple server containers: Can be run in parallel behind load balancer
- Multiple worker containers: Can be run in parallel; each processes independent tasks from Redis queue
- No state sharing between containers beyond PostgreSQL and Redis

**Container Artifact Management:**

- Single image artifact deployed multiple times with different configurations
- Server instance: `docker run --build-arg ... Dockerfile:latest`
- Worker instance: `docker run --build-arg ... worker/Dockerfile:latest`
